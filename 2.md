# Phase 2 — Admin Panel Upgrade: Approval UX, Rejection Reason, Provider Data, Verification Switches & User Funnel Analytics

---

## Phase Goal

This phase rebuilds the admin panel experience on top of the existing dashboard without replacing the core Django template or auth system. It adds five major admin-facing capabilities:

1. **Approval UX Fix** — After approving or rejecting a contact request or service, stay on the current page (no redirect). JavaScript handles the update inline using AJAX. The approval card is updated in-place.
2. **Rich Provider Data in Approvals** — The approval card must show the provider's Telegram username, phone number, and all uploaded photos before the admin decides.
3. **Rejection with Mandatory Reason** — When rejecting a service registration, admin must write why. The reason is saved to `ServiceProfile.rejection_reason`. The bot sends the rejection reason to the provider with an inline "Register Again" button. The provider's service profile data is deleted. The provider's role reverts to CUSTOMER. The provider sees the rejection reason clearly.
4. **Admin Toggle Switches: Personal Verification, Tested Badge, Admin-Forced Visibility** — Three new toggle switches in the admin panel for each provider: (a) `is_verified` on `TelegramUser`, (b) `admin_tested_badge` on `TelegramUser` (added in Phase 1), (c) `admin_forced_hidden` on `ServiceProfile` (added in Phase 1). Each toggle triggers an AJAX call and updates instantly.
5. **User Funnel Analytics** — Admin dashboard shows how many users ran `/start`, how many passed the policy quiz, how many are mid-registration, and provides a filtered list to text users who started registering but stopped 2-3 days ago. Includes a cron-triggered mass-reminder function for those users.

## Background

The admin dashboard at `/dashboard/admin/` renders a Django HTML template (`templates/adminpanel/dashboard.html`). It currently uses full-page form POSTs for approve/reject actions — which redirects to the dashboard home. The `approvals/contact_workflow.py` module handles notifications to providers and customers. The `bot/service_notifications.py` module handles service status notifications. The `TelegramUser` model stores `is_verified` and the new `admin_tested_badge` field. The `ServiceProfile` model stores the new `admin_forced_hidden` and `rejection_reason` fields. The `BotRegistrationSession` model stores the current registration step for each user. The `bot/services.py` `TelegramBotService` is used to send all Telegram messages.

## Detailed Requirements

### 1. Approval UX — Stay on Page

Currently, `dashboard_contact_action` and `dashboard_service_action` redirect to `redirect("adminpanel-dashboard")` after action. This causes the page to scroll to the top.

**New behavior:** The dashboard page JavaScript intercepts every approve/reject form submission, submits via `fetch()` to existing AJAX endpoints (`/api/admin/contact/approve/`, `/api/admin/contact/reject/`, `/api/admin/service/approve/`, `/api/admin/service/reject/`), and updates the card DOM in-place on success.

The existing AJAX API endpoints in `adminpanel/views.py` (`approve_contact`, `reject_contact`, `approve_service`, `reject_service`) already exist and use DRF token authentication. The dashboard must send the DRF token in these AJAX calls.

To get the admin's DRF token in the dashboard template: add `admin_token` to the dashboard context. This is the DRF auth token for the logged-in admin's linked TelegramUser (if available), or it is omitted for pure staff users. For pure staff users, the AJAX calls use Django session cookies instead. The AJAX endpoints must support BOTH session authentication and token authentication (already configured in `REST_FRAMEWORK.DEFAULT_AUTHENTICATION_CLASSES`).

**Implementation:**

In `templates/adminpanel/dashboard.html`, add a `<script>` block that:
1. Intercepts form submit events on all `.contact-approve-form`, `.contact-reject-form`, `.service-approve-form`, `.service-reject-form` elements using event delegation.
2. For reject actions on services: shows an inline modal asking for `rejection_reason` text (required, min 10 chars). The modal must be a native HTML `<dialog>` element — no external library.
3. On AJAX success: updates the card's status badge, disables the approve/reject buttons on that card, and shows a success indicator.
4. On AJAX failure: shows an inline error message on the card. Does not reload the page.
5. If a CSRF token is needed for session-auth AJAX calls, reads it from the cookie (`csrftoken`) and adds it to the `X-CSRFToken` header.

**API endpoint changes:**

`reject_service` (POST `/api/admin/service/reject/`) must accept an optional `rejection_reason` field via `AdminServiceActionSerializer`. If not provided (or empty string), the action is rejected with HTTP 400: `{"success": false, "error": "rejection_reason is required when rejecting a service."}`.

`reject_contact` does NOT require a reason (contact rejections are system-level).

### 3. Rejection with Mandatory Reason

When admin rejects a service registration:

**Server-side (Django):**

In `reject_service` view:
1. Receive `service_id` and `rejection_reason` (required, non-empty, min 10 chars, max 1000 chars).
2. Set `service.rejection_reason = rejection_reason`.
3. Set `service.approval_status = REJECTED`.
4. Save service.
5. Atomically within the same transaction:
   - Delete all `ServicePhoto` records for this service.
   - Delete all `ServicePrice` records for this service.
   - Delete the `ServiceProfile` itself.
   - Set the provider's `TelegramUser.role = CUSTOMER`.
   - Delete the provider's `BotRegistrationSession` if one exists.
6. After the transaction, use `transaction.on_commit()` to call a new function `queue_service_rejection_with_reason(provider_telegram_id, rejection_reason)`.

**`queue_service_rejection_with_reason(provider_telegram_id, rejection_reason)`** (new function in `bot/service_notifications.py` or a new module):
- Send a Telegram message to the provider's chat ID:
  ```
  ❌ Your service registration was not approved.

  Reason: {rejection_reason}

  You can start the registration process again when you are ready.
  ```
- Include an inline button: `[📝 Register Again]` with callback_data `"registration:create_service"` (this is the existing callback that starts a new registration).
- This reuses `TelegramBotService.send_text()` with `build_register_again_keyboard()` (new keyboard builder in `TelegramBotService`).

**Bot-side:** The "Register Again" button triggers the existing `CREATE_SERVICE_CALLBACK = "registration:create_service"` handler in `bot/handler_modules/registration.py`. This is already implemented. No change needed.

**Dashboard-side:** After AJAX rejection, remove or gray out the service card in the DOM.

### 4. Admin Toggle Switches

Add three new API endpoints to `adminpanel/views.py`:

#### `POST /api/admin/provider/<int:provider_id>/toggle-verified/`
- Finds `TelegramUser` by `id=provider_id`.
- Toggles `TelegramUser.is_verified` (True → False, False → True).
- Saves using `update_fields=["is_verified", "updated_at"]`.
- Returns `{"success": true, "is_verified": <new_value>}`.
- Permission: admin TelegramUser or Django staff.

#### `POST /api/admin/provider/<int:provider_id>/toggle-tested/`
- Finds `TelegramUser` by `id=provider_id`.
- Toggles `TelegramUser.admin_tested_badge`.
- Saves using `update_fields=["admin_tested_badge", "updated_at"]`.
- Returns `{"success": true, "admin_tested_badge": <new_value>}`.
- Permission: admin TelegramUser or Django staff.

#### `POST /api/admin/service/<int:service_id>/toggle-admin-visibility/`
- Finds `ServiceProfile` by `id=service_id`.
- Toggles `ServiceProfile.admin_forced_hidden`.
- Saves using `update_fields=["admin_forced_hidden", "updated_at"]`.
- Returns `{"success": true, "admin_forced_hidden": <new_value>}`.
- Permission: admin TelegramUser or Django staff.

All three endpoints use `@api_view(["POST"])` and `@permission_classes([IsAuthenticated])`.

**Dashboard template:** For each provider row in the providers tab and each service card in the services tab, add three toggle switches (HTML `<input type="checkbox">` styled as a pill switch using CSS only — no external library). The JS sends a POST request on toggle change and updates the switch state optimistically. On failure, reverts the toggle and shows an error.

**Auto-approve toggle:** The existing `AdminSettings.auto_approve_requests` field controls auto-approval. The dashboard must show this as a visible toggle switch. Update it via the existing `PATCH /api/admin/settings/` endpoint. The toggle state is loaded from the `settings` context variable already available in the dashboard.

Add URL patterns for the three new endpoints to `marketplace/urls.py`.

### 5. User Funnel Analytics

Add these statistics to the dashboard context in `admin_dashboard()`:

```python
from bot.models import BotRegistrationSession
from django.utils import timezone
from datetime import timedelta

# Users who have ever sent /start (created a TelegramUser)
total_started = TelegramUser.objects.count()

# Users who passed the policy quiz
total_policy_passed = TelegramUser.objects.filter(
    policy_accepted_at__isnull=False
).count()

# Users currently mid-registration (session exists, not completed/cancelled)
mid_registration_states = [
    BotRegistrationSession.State.SELECT_ROLE,
    BotRegistrationSession.State.PROVIDER_PHONE,
    BotRegistrationSession.State.SECONDARY_PHONE,
    BotRegistrationSession.State.CATEGORY,
    BotRegistrationSession.State.TITLE,
    BotRegistrationSession.State.DESCRIPTION,
    BotRegistrationSession.State.LOCATION,
    BotRegistrationSession.State.PRICES,
    BotRegistrationSession.State.PHOTOS,
    BotRegistrationSession.State.SUBMIT,
]
total_mid_registration = BotRegistrationSession.objects.filter(
    state__in=mid_registration_states
).count()
```

Add to context dict under `"funnel"` key.

Display in the stats grid in the template as three new stat cards: "👤 Total Users", "✅ Policy Passed", "⏳ Mid-Registration".

**Filterable Reminder List:**

Add a new tab "⏳ Reminders" to the dashboard tabs. This tab shows a table of users who:
- Have a `BotRegistrationSession` in a mid-registration state (not COMPLETED, not CANCELLED)
- AND their `last_interaction_at` (added in Phase 1) is between 2 and 3 days ago (i.e., `now - 3 days < last_interaction_at <= now - 2 days`)

Query:
```python
now = timezone.now()
reminder_cutoff_start = now - timedelta(days=3)
reminder_cutoff_end = now - timedelta(days=2)

reminder_users = BotRegistrationSession.objects.filter(
    state__in=mid_registration_states,
    updated_at__gte=reminder_cutoff_start,
    updated_at__lte=reminder_cutoff_end,
).select_related("...")  # Note: BotRegistrationSession has telegram_user_id, not a FK
```

Note: `BotRegistrationSession` has `telegram_user_id` as a `BigIntegerField`, not a FK. To join: fetch the session IDs, then look up `TelegramUser` records.

Display columns: Telegram ID, Username, Current State, Last Updated.

Add a "📨 Send Reminder" button per row that sends a Telegram message to that user via AJAX.

**New endpoint `POST /api/admin/send-registration-reminder/`:**
- Body: `{"telegram_user_id": <int>}`
- Finds the `TelegramUser` by `telegram_id`.
- Finds their active `BotRegistrationSession`.
- If session and user exist, sends a bot message:
  ```
  👋 You started registering your service profile but haven't finished yet.
  
  Tap below to continue where you left off:
  ```
  With inline button `[📋 Continue Registration]` → callback_data `"registration:create_service"`.
- Returns `{"success": true}` or `{"success": false, "error": "..."}`.

**Mass reminder endpoint `POST /api/admin/send-mass-reminders/` (cron-triggered):**
- No body needed (admin-only, verified via Django auth).
- Runs the same filter: mid-registration users with `updated_at` between 2 and 3 days ago.
- Sends the reminder message to each.
- Returns `{"success": true, "sent_count": N, "failed_count": M}`.
- This endpoint is designed to be called by a cron job (e.g., once per day at 09:00 Africa/Addis_Ababa).
- Rate limit: max 200 reminders per call. Log each send attempt.
- Must be idempotent: sending a reminder a second time in the same day is acceptable; no deduplication is required for Phase 2.

Add URL patterns for both new endpoints to `marketplace/urls.py`.

## API Requirements

### Modified Endpoints

**`POST /api/admin/service/reject/`**
- Now requires `rejection_reason` (string, min 10 chars, max 1000 chars) in body.
- Returns 400 if missing or too short.
- On success: returns `{"success": true, "message": "Service rejected and provider notified."}`.

### New Endpoints

**`POST /api/admin/provider/<int:provider_id>/toggle-verified/`**
- Auth: Django session or DRF token, must be staff/superuser or admin TelegramUser.
- Response 200: `{"success": true, "is_verified": <bool>}`
- Response 404: `{"success": false, "error": "Provider not found."}`

**`POST /api/admin/provider/<int:provider_id>/toggle-tested/`**
- Same auth as above.
- Response 200: `{"success": true, "admin_tested_badge": <bool>}`

**`POST /api/admin/service/<int:service_id>/toggle-admin-visibility/`**
- Same auth as above.
- Response 200: `{"success": true, "admin_forced_hidden": <bool>}`

**`POST /api/admin/send-registration-reminder/`**
- Body: `{"telegram_user_id": <int>}` (the DB ID of TelegramUser, not telegram_id)
- Response 200: `{"success": true}`
- Response 404: `{"success": false, "error": "User not found or no active session."}`

**`POST /api/admin/send-mass-reminders/`**
- No body.
- Response 200: `{"success": true, "sent_count": N, "failed_count": M}`
- Response 403 if not admin.

### Error Responses
All endpoints return `{"success": false, "error": "<message>"}` on failure with appropriate HTTP status code.
