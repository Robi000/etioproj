# Phase 4 — Mini App Frontend: Complete UI, Customer Service Request Flow, Badge System, Provider-Customer Skip

---

## Phase Goal

This phase builds the Telegram Mini App frontend from scratch. The current `miniapp_landing` view in `miniapp/views.py` returns a "Coming Soon" placeholder page. This phase replaces it with a fully functional, premium-quality single-page application that handles:

1. **Complete Mini App Frontend** — Swipe/grid discovery, service cards, contact request creation, saved services, profile view. Built in HTML + vanilla CSS + vanilla JavaScript (no React, no Vue, no framework). Served from the existing Django backend.

2. **Create Service → Customer Skip to Category** — When a user taps "Create Service" and then "Customer", instead of going through role selection, skip directly to step 3 (category selection from the provider's registered categories). This is a bot-side fix.

3. **Customer Price Minimum + Average Display** — When a customer enters a price expectation (in the contact request flow), the minimum must be 1000 ETB. The system must show the average price for that service category.

4. **Badge System** — Based on `likes_count` on `TelegramUser` (added Phase 1) / `ServiceProfile` (added Phase 1), providers earn progressive emoji badges. Admin can also grant a manual verified badge. Both are displayed on service cards.

5. **Verification & Tested Badges in UI** — Show `is_verified` (checkmark badge) and `admin_tested_badge` (star badge) on service cards in the Mini App.

6. **Provider/Customer Card Flow** — Swipe interface with rich cards, displaying all service data, photo carousel, badges, price, and contact request button.

---

## Background

The Mini App is a Telegram WebApp served at the domain configured in `TELEGRAM_MINI_APP_URL`. The existing backend at `/` (the `miniapp_landing` view) serves HTML. The Mini App authenticates using Telegram `initData` and the `POST /api/auth/telegram/` endpoint, which returns a DRF token. This token is used for all subsequent API calls. All discovery APIs (`/api/discovery/grid/`, `/api/discovery/swipe/`) are already implemented and returning data. The swipe like/dislike APIs exist. The contact request API exists. The service categories API does not yet exist — a new lightweight endpoint is needed. The `TelegramUser` model has `is_verified`, `admin_tested_badge`, `likes_count`. The `ServiceProfile` has `likes_count`. The `SwipeHistory` model tracks swipes. The `SavedServiceRequest` model was added in Phase 3.

### 1. Mini App Architecture

The Mini App is a single HTML file served by `miniapp_landing()` in `miniapp/views.py`.

Replace the current placeholder HTML with a full application HTML file. The view reads from `templates/miniapp/app.html` (create this file).

```python
def miniapp_landing(request: HttpRequest) -> HttpResponse:
```

**Technology stack:**
- HTML5 semantic structure
- Vanilla CSS (embedded in `<style>` tag or `<link>` to `/static/miniapp/app.css`)
- Vanilla JavaScript (ES2020+, no transpiler, no bundler)
- Telegram WebApp SDK: `https://telegram.org/js/telegram-web-app.js`

**Design language:**
- Dark mode by default (Telegram Mini App inherits theme from Telegram)
- Use `window.Telegram.WebApp.colorScheme` to detect theme
- Color palette: deep navy background (`#0d1117`), teal accent (`#14b8a6`), amber highlight (`#f59e0b`)

- Rounded corners: 16px on cards, 999px on pills
- Typography: Inter font, 15px base, bold headings
- Micro-animations: CSS transitions on all interactive elements (150ms ease)


**Screen routing:**
The `screen` query parameter (already supported by `build_mini_app_url(screen=...)` in `bot/services.py`) controls the initial screen. Supported values: `"my-service"`, `"saved"`, `"swipe"`, `"grid"`.

### 2. App Load Sequence

On page load:
1. Initialize `window.Telegram.WebApp.ready()`.
2. Extract `initData` from `window.Telegram.WebApp.initData`.
3. POST to `/api/auth/telegram/` with `{"init_data": initData}`.
4. On success: store `token` and `user` in state.
5. Check `user.policy_accepted_at`: if null, show policy gate screen.
6. If policy accepted: navigate to home screen.
7. Home screen: two large buttons — "🔍 Find a Service" (→ swipe screen) and "📋 My Service" (→ my service screen).

**Policy Gate Screen:**
If `user.policy_accepted_at` is null:
- Show a screen: "To use this marketplace, please open the bot and complete the policy review."
- Show a button: "📱 Open Bot" that deep-links to `https://t.me/{BOT_USERNAME}?start=policy`.
- The bot username is embedded in the HTML as a template variable: `{{ TELEGRAM_BOT_USERNAME }}`.
- The `miniapp_landing` view passes this to the template via context:
  ```python
  return render(request, "miniapp/app.html", {"bot_username": settings.TELEGRAM_BOT_USERNAME})
  ```
- The JS reads this from a `<meta name="bot-username" content="{{ bot_username }}">` tag.

### 3. Swipe Discovery Screen

#### UI
- One service card at a time, centered on screen.
- Card shows: first photo (full-bleed at top), provider name + badges, service title, category chip, price range, city.

- `provider_pending`: "⏳ Waiting for provider..."

- `approved` or `auto_approved`: show "✅ Contact: {contact_value}"

### 6. Contact Request Flow with Price Minimum

When the customer taps "Contact Provider":

1. Check if customer has GPS location (from `user.has_customer_location` via `/api/me/`). If not: show inline prompt to open bot and share location.
2. Show a modal/bottom sheet: "Set your budget for this service."
3. Input field: minimum 1000 ETB. Show validation error if below 1000: "Minimum budget is 1000 ETB."
4. Show below the input: "Average price for {category}: {avg_half_day} / {avg_full_day} / {avg_night} ETB." (fetch from new `GET /api/services/category-avg-price/?category_id=N` endpoint).
5. On confirm: POST to `/api/contact-request/` with `{"service_id": ...}`. (The budget field is for UX only in Phase 4 — it is NOT sent to the server. Phase 5 may use it for ranking. For now, just enforce min 1000 in UI and show average, then proceed with the regular contact request creation.)
6. Show contact request status.

#### New Backend Endpoint

**`GET /api/services/category-avg-price/`**
- Query param: `category_id` (required).
- Computes `AVG(amount)` for each `price_type` among all approved, active service prices in that category.
- Returns:
  ```json
  {
    "success": true,
    "category_id": 1,
    "category_name": "Cleaner",
    "averages": {
      "half_day": "1250.00",
      "full_day": "2100.00",
      "night": "1800.00"
    }
  }
  ```
- Permission: `AllowAny` (public info).
- Uses `ServicePrice.objects.filter(service__category_id=category_id, service__approval_status=APPROVED).values("price_type").annotate(avg_amount=Avg("amount"))`.
- One DB query. No caching needed for Phase 4.

Add to `services/views.py` and `services/urls.py`.

### 7. Create Service → Customer → Category Skip (Bot Fix)

Current behavior for `role == ROLE_CUSTOMER`:
```python
session.state = BotRegistrationSession.State.CATEGORY
```

This is already implemented but the "Customer" role is for customers browsing — it does NOT need service category registration.

Wait — re-reading the requirement: "when the user tap on create service then customer... it directly send it to step three select your catagory from the provider choices."

This means: when a CUSTOMER (someone who wants to hire) taps "Create Service" → the bot shows the SERVICE CATEGORIES that are available (provider categories to choose from for browsing), not a registration form. This is a redirect to the swipe/grid screen filtered by category.

**Correct interpretation:**
- When a user taps "Create Service" callback in the bot, check their role.
- If they are a customer (role = CUSTOMER and no ServiceProfile), skip provider registration entirely.
- Instead, send: "Which service are you looking for?" + the category selection keyboard.
- After they pick a category, open the Mini App on the swipe screen filtered by that category: `build_mini_app_url(screen="swipe")` with `?category_id=N` appended.
- The "Create Service" callback is handled in `bot/handler_modules/registration.py`. Add this customer-branch check at the top of the handler.

Add a new bot keyboard `build_customer_category_keyboard()` in `TelegramBotService` that shows categories with callback_data `"customer:browse:category:<category_name>"`.

Add callback handler `can_handle_callback()` for `"customer:browse:"` prefix.

Handler: find the category by name, build the Mini App URL with the category filter, send the Mini App button.
