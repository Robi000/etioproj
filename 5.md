# Phase 5 — Discovery Intelligence: Ranked Swipe Algorithm, Exploration Injection & Performance Optimization

---

## Phase Goal

This phase replaces the current simple proximity + creation date sort in `get_discovery_services()` (in `matching/views.py`) with the full ranked discovery algorithm designed for the marketplace. It must not break any existing API contract or require frontend changes. The change is purely in the backend discovery service function.

Additionally, this phase handles all performance optimizations, missing indexes, and any remaining cleanup from prior phases.

1. **Ranked Discovery Feed** — Implement a multi-factor ranking score for each candidate service, using proximity, quality (likes), price competitiveness, freshness, and demand-fairness signals.

2. **Exploration Injection** — Every 5th card in the feed must be a random "exploration card" — a randomly selected provider from the approved pool who scored below the top 20% in the ranked list. This ensures new providers get discovery exposure even if their score is low.

3. **Demand-Fairness Signal** — Providers who received many recent contact requests are deprioritized to give lower-demand providers a chance to be discovered.

4. **Precomputed Ranking Score** — The ranking computation must not block the API request. A `ranking_score` field is precomputed and stored on `ServiceProfile`. A cron job recomputes scores periodically. The live query uses the stored score.

5. **Database Performance Audit** — Audit all existing queries across every app for N+1 problems, missing indexes, and over-fetching. Fix any discovered issues.

6. **`provider_pending` Status in Contact Limit Count** — The status `PROVIDER_PENDING` must count towards the customer's daily contact limit. This is a clarification of Phase 1 logic: update `evaluate_contact_request_creation()` to count `PROVIDER_PENDING` contact requests in the daily request count.

---

## Background

The current `get_discovery_services()` in `matching/views.py`:
1. Fetches all approved, visible, non-banned services from DB.
2. Computes Haversine distance from customer GPS.
3. Filters by radius (from `AdminSettings.default_radius`).
4. Sorts by `(distance_km, created_at)`.
5. Returns the sorted list.

This is correct but naive. It loads all matching services into Python memory, computes distance in Python, then sorts in Python. For large datasets, this does not scale. For the current SQLite dataset, it is acceptable but should be optimized.

The new algorithm adds a multi-factor composite score that replaces the simple sort. The score is precomputed and cached in a new `ServiceProfile.ranking_score` field.

All existing API consumers (Mini App, swipe endpoints) receive the same response schema — only the ordering changes.

## Detailed Requirements

### 1. Ranking Algorithm

#### New File: `matching/ranking.py`

The ranking score is a composite value between 0 and 100 (float). Higher = better.

#### Score Components

| Component | Weight | Description |
|-----------|--------|-------------|
| Proximity | 35% | Closer providers score higher |
| Quality | 30% | Based on `likes_count` (logarithmic scale) |
| Price competitiveness | 15% | Relative to category average |
| Demand-fairness | 10% | Fewer recent contact requests = higher score |
| Freshness | 10% | Recently created/updated services score higher |

#### Proximity Score (35 points max)

```
proximity_score = max(0, 1 - (distance_km / max_radius_km)) * 35
```

Where `max_radius_km = AdminSettings.default_radius`.

If `distance_km` is None (no GPS), `proximity_score = 0`.

Providers with no GPS location but who match by city_text get a default `distance_km = max_radius_km * 0.75` (placed at 75% of max radius, i.e., disadvantaged but not zeroed out).

#### Quality Score (30 points max)

```
quality_score = min(1.0, math.log1p(service.likes_count) / math.log1p(200)) * 30
```

`log1p` prevents zero-division and compresses the scale. At 200 likes, quality score = 30 (max).

#### Price Competitiveness Score (15 points max)

```
category_avg = (precomputed or query) category average price for primary price_type

if service_avg_price <= category_avg:
    price_score = 15  # at or below average
else:
    ratio = category_avg / service_avg_price  # ratio < 1 for above-average price
    price_score = max(0, ratio * 15)
```

`service_avg_price`: average of all prices set on this service profile.

If no prices set: `price_score = 7.5` (neutral).

Category averages are computed once per ranking run and passed to the function as a `dict[category_id → float]`.

#### Demand-Fairness Score (10 points max)

```
recent_requests = ContactRequest.objects.filter(
    provider=service.provider,
    created_at__gte=now - timedelta(days=7),
).count()

demand_score = max(0, 10 - min(10, recent_requests))
```

Providers with 0 recent requests get 10 points. Providers with ≥10 recent requests get 0 points.

This is computed once per ranking run using a batch query for all candidate providers.

#### Freshness Score (10 points max)

```
days_old = (now - service.created_at).days
freshness_score = max(0, 10 - (days_old / 30))  # loses 1 point per 30 days, max 10
```

Services newer than 1 day get 10 points. Services older than 300 days get 0.

#### Total Score

```python
def compute_ranking_score(
    service: ServiceProfile,
    distance_km: float | None,
    category_avg_prices: dict[int, float],
    recent_request_counts: dict[int, int],  # provider_id → count
    settings: AdminSettings,
) -> float:
    proximity = proximity_score(distance_km, settings.default_radius)
    quality = quality_score(service.likes_count)
    price = price_score(service, category_avg_prices.get(service.category_id))
    demand = demand_score(recent_request_counts.get(service.provider_id, 0))
    freshness = freshness_score(service.created_at)
    return round(proximity + quality + price + demand + freshness, 4)
```

Algorithm:
1. Fetch all approved, visible, not-hidden services with their provider, category, prices, and swipe_history in a single queryset with `select_related` and `prefetch_related`.
2. Compute batch data:

4. Bulk update: `ServiceProfile.objects.bulk_update(services_list, ["ranking_score", "last_ranked_at"], batch_size=100)`.

Returns `{"success": true, "updated_count": N}`.

Designed to be called every 4 hours via system cron.

### 3. Updated Discovery Service Function

#### `get_discovery_services()` in `matching/views.py`

**New algorithm:**

Step 1: Proximity pre-filter using bounding box (pure SQL).

If customer has lat/lon, compute a bounding box:
```python
lat_delta = radius_km / 111.32
lon_delta = radius_km / (111.32 * math.cos(math.radians(float(customer_lat))))

lat_min = customer_lat - lat_delta
lat_max = customer_lat + lat_delta
lon_min = customer_lon - lon_delta
lon_max = customer_lon + lon_delta
```

Filter queryset: `.filter(latitude__gte=lat_min, latitude__lte=lat_max, longitude__gte=lon_min, longitude__lte=lon_max)`.

This is a DB-level rectangular pre-filter that eliminates most out-of-range providers before Python processes them. Haversine is still computed in Python for the precise distance, but only on the ~10-20% of providers that survive the bounding box.

Step 2: Fetch candidate services (with bounding box filter applied). Use `select_related("provider", "category").prefetch_related("prices")`. Do NOT prefetch `photos` at this stage (photos are fetched only for the final ranked card).

Step 3: For each candidate, compute Haversine distance and apply exact radius filter.

Step 4: Compute live scores:
- Use precomputed `service.ranking_score` as base.
- Apply a live proximity adjustment: `live_score = service.ranking_score + proximity_adjustment(distance_km, settings.default_radius)`.
- The `proximity_adjustment` is a delta applied ON TOP of the stored base score to make proximity real-time.

Why this works: the stored `ranking_score` includes a default proximity of 0 (computed without GPS). The live adjustment adds the actual proximity component. This avoids recomputing the full score on every request.

Precisely:
- Stored score = quality + price + demand + freshness (35 + 30 = max 65 points from non-proximity components)
- Actually this conflicts with the 35-point proximity being part of the stored score.
- Correct design: stored `ranking_score` = quality + price + demand + freshness (max 65 points, computed without proximity). Live, add proximity (max 35 points). Total = 0-100.
- Update `compute_ranking_score()` to NOT include proximity. The score stored is 0-65.
- Live: `live_score = service.ranking_score + compute_proximity_score(distance_km, settings.default_radius)`.

Step 5: Sort by `live_score DESC`.

Step 6: Inject exploration cards (see Section 4).

Step 7: Return the final list.

**Backward compatibility:** The function signature and return type do not change. The `service_rows` list format remains identical. Only ordering changes.

### 4. Exploration Injection

In `matching/ranking.py`, add:

```python
def inject_exploration_cards(
    ranked_list: list[dict],
    all_candidates: list[dict],
    injection_interval: int = 5,
    top_threshold_fraction: float = 0.2,
) -> list[dict]:
    """
    Every `injection_interval`-th position is replaced by a random
    low-ranking provider (from the bottom 80% by ranking score).
    This gives new/low-visibility providers discovery exposure.
    """
    if len(ranked_list) < injection_interval * 2:
        return ranked_list  # Not enough cards for meaningful injection

    top_cutoff = max(1, int(len(ranked_list) * top_threshold_fraction))
    exploration_pool = ranked_list[top_cutoff:]  # Bottom 80%

    if not exploration_pool:
        return ranked_list

    result = list(ranked_list)
    injection_positions = list(range(injection_interval - 1, len(result), injection_interval))

    random.shuffle(exploration_pool)
    exploration_iter = iter(exploration_pool)

    for pos in injection_positions:
        try:
            exploration_card = next(exploration_iter)
            # Only inject if this position is not already an exploration card
            result[pos] = {**exploration_card, "_is_exploration": True}
        except StopIteration:
            break  # Ran out of exploration cards

    return result
```

**Important:** Exploration injection modifies the `ranked_list` in place (via copy). It does NOT add new cards — it replaces the card at every 5th position. If the replaced card was already a top card, it is simply moved to a later position or dropped.

The `_is_exploration` flag is internal only and must be removed before building the response (or ignored during serialization since `DiscoveryServiceCardSerializer` has fixed fields).

For the `discovery_swipe` endpoint (which returns one card at a time): the caller calls the function repeatedly. The exploration injection still applies to the whole list, then the first card is returned. On the next call, the next card is returned. This is stateless — each call recomputes from scratch. Exploration injection is non-deterministic (uses `random`) which is acceptable.

## Appendix: Discovery Algorithm v1 Score Summary

| Component | Weight | Formula | Max Points |
|-----------|--------|---------|------------|
| Proximity | 35% | `max(0, 1 - dist/radius) * 35` | 35 |
| Quality | 30% | `min(1, log1p(likes)/log1p(200)) * 30` | 30 |
| Price | 15% | `15` if at/below avg, `ratio * 15` if above | 15 |
| Demand-Fairness | 10% | `max(0, 10 - min(10, recent_reqs))` | 10 |
| Freshness | 10% | `max(0, 10 - days_old/30)` | 10 |
| **Total** | **100%** | — | **100** |

**Stored vs. live:**
- Stored `ranking_score` = quality + price + demand + freshness = 0–65
- Live score = stored + proximity = 0–100

**Exploration Injection:**
- Injection interval: every 5th position
- Exploration pool: bottom 80% by live_score
- Injection is non-deterministic (random selection from pool)
- Pool is shuffled once per request, then popped sequentially

**Grade Period Note:**
- Providers with `likes_count = 0` and newly created services receive `quality_score = 0` and `freshness_score = 10`. Their base score is `0 + price_score + demand_score + 10`. They still appear in the feed via exploration injection even if their proximity ranks them last.
