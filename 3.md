# Phase 3 — Bot Intelligence: Provider Penalty System, GPS Enforcement, Proactive Request Alerts, Saved Requests, Customer Survey & Advertisement Broadcast

---

## Phase Goal

This phase implements all Telegram bot behavioral intelligence that operates behind the scenes — system-level rules that enforce fair provider behavior, protect customers, and keep engagement high.

1. **Provider Penalty & Ban System** — Too many contact request denials (or timeouts) penalizes providers. The ratio of denials to total requests is calculated after a grace period of 10 accepted requests. Above 75% denial ratio triggers a temporary ban. A second violation extends the ban. Uses `denial_count` and `penalty_until` fields from Phase 1.

2. **Provider Request Timeout as Denial** — Contact requests not answered within 24 hours are automatically counted as a denial. A cron job detects these and applies the denial logic.

3. **Proactive Pending Request Alert** — When a provider interacts with the bot for any reason (any message or callback), if they have any unanswered contact request (status = PROVIDER_PENDING), the bot reminds them with an inline button to act on it.

4. **Customer GPS Collection** — After policy acceptance, when a customer tries to do anything (swipe, browse), if they have no GPS data in `TelegramUser.customer_latitude/longitude` (added Phase 1), they are prompted to share their location first. GPS must be within Ethiopia's bounding box. If GPS is too far from Addis Ababa (or out of Ethiopia range), notify with inline button to re-share.

5. **Saved Service Requests** — Customers can save up to 3 provider service profiles for later. A new `SavedServiceRequest` model stores these. Bot and Mini App both support this flow.

6. **Post-Service Customer Survey** — 2 days after a contact request is approved, the customer receives a survey asking if they received the service. If "No", a follow-up asks why (with inline buttons for 6 predefined reasons).

7. **Advertisement Broadcast** — Admin can trigger a broadcast that sends 3 random photos from approved services to all customers. A cron-triggered endpoint handles the broadcast. Sends first 3 photos from top-performing (most liked + approved) services.

8. **Photo Change Re-Approval** — If an approved provider changes any of their service photos, the photo change must go through admin approval before becoming visible.

9. **Privacy Settings Warning** — Just after provider registration submission and before sending the username, the bot must check if the provider's Telegram privacy settings might block message forwarding. If so, send a warning with instructions.

#### New Model: `ProviderDenialLog` in `services/models.py`

Meta: `db_table = "provider_denial_logs"`, index on `service`, index on `created_at`.

#### Penalty Calculation

The penalty is evaluated every time a new denial event is recorded.

Grace period: the first 10 contact requests received by a provider are exempt from penalty calculation (regardless of denial ratio). Only start calculating ratio after 10 total requests.

Total requests = `ContactRequest.objects.filter(provider=service.provider).count()`

Denial ratio = `service.denial_count / total_requests`

If `total_requests >= 10` and `denial_ratio > 0.75`:
- Check if provider has had a penalty before (i.e., `service.penalty_until` was ever set):
  - If first penalty: `service.penalty_until = timezone.now() + timedelta(days=7)`
  - If subsequent penalty: `service.penalty_until = timezone.now() + timedelta(days=15)`
- Also set `service.visibility_status = VisibilityStatus.OFF` to immediately hide them.
- Save with `update_fields=["denial_count", "penalty_until", "visibility_status", "updated_at"]`.

Track whether the provider has had a prior penalty by checking if `ProviderDenialLog` for this service has records AND `penalty_until` has ever been set. Simpler approach: add `penalty_count` field (PositiveIntegerField, default=0) to `ServiceProfile` in a new migration.

Wait — do NOT add more fields without good reason. Instead: check `ProviderDenialLog.objects.filter(service=service, reason="timeout").count() > 0` as a proxy for "prior penalty". Actually the cleaner approach: add a single `prior_penalty_count` IntegerField(default=0) to `ServiceProfile` in this phase's migration. Increment it each time a penalty is applied.

Migration: new file `services/migrations/XXXX_add_denial_log_and_penalty_count.py`.

#### Denial Increment Points

In `bot/handler_modules/contact_requests.py`, `reject_contact_request()`:
```python
service = contact_request.service
if service:
    service.denial_count += 1
    service.save(update_fields=["denial_count", "updated_at"])
    ProviderDenialLog.objects.create(
        service=service,
        reason="manual_reject",
        contact_request=contact_request,
    )
    _evaluate_and_apply_penalty(service)
```

#### Provider Warning on High Denial Count

Every time a provider interacts with the bot AND `service.denial_count >= 5`:
- Send a warning message once per session (not on every single message — use a flag in `BotRegistrationSession.data` or a simple in-memory check per update):
  - "⚠️ Warning: You have denied {denial_count} contact requests. If your denial rate exceeds 75% after 10 requests, you will be temporarily suspended. To avoid this, toggle your visibility Off when you are unavailable."
- Include inline button: "Toggle Offline" → `callback_data="profile:go_offline"`.
- The warning is sent as a separate message AFTER the provider's normal response.

#### Provider Visibility Switch

The provider can toggle their visibility to "Off" to avoid timeouts and denials when unavailable. This is the existing "Go Offline" flow in `bot/handler_modules/registration.py`. No change needed to the toggle logic. The warning message directs them to use it.

### 2. 24-Hour Timeout as Denial

#### Cron Endpoint: `POST /api/admin/process-timeouts/`

- Auth: Django staff or admin TelegramUser.
- Finds all `ContactRequest` records where:
  - `status = PROVIDER_PENDING`
  - `created_at < now - 24 hours`
- For each:
  1. Set `status = PROVIDER_REJECTED` (existing status choice).
  2. Increment `ServiceProfile.denial_count` for the provider.
  3. Create `ProviderDenialLog(service=service, reason="timeout", contact_request=contact_request)`.
  4. Call `_evaluate_and_apply_penalty(service)`.
  5. Queue customer rejection message: `queue_customer_rejection_message(contact_request.id)`.
- Returns `{"success": true, "processed_count": N}`.
- Wrap in `transaction.atomic()` per record (or per batch with appropriate error handling).
- Log: `logger.info("event=timeout_denial_applied contact_request_id=%s provider_id=%s")`.

Designed to be called by a system cron job every hour.

### 3. Proactive Pending Request Alert

In `bot/handler_modules/start.py` `handle_start_command()` AND in `bot/dispatcher.py` (main dispatch function, after routing), after resolving the provider's `TelegramUser`:

Check: if provider has any `ContactRequest` with `status = PROVIDER_PENDING`:
```python
pending_request = ContactRequest.objects.filter(
    provider__telegram_id=context.telegram_user_id,
    status=ContactRequest.Status.PROVIDER_PENDING,
).select_related("service", "customer").first()
```

If found AND the provider's current action is NOT already responding to this request (i.e., `callback_data` does not start with `"contact:"`):
- Send a separate message:
  ```
  🔔 You have a pending service request from {customer.get_display_name()}!
  
  Service: {service.title}
  
  Please respond:
  ```
- With inline keyboard: Accept / Reject buttons using the existing `build_contact_request_decision_keyboard(contact_request.id)`.

This check must be efficient: add an index on `(provider, status)` in `ContactRequest.Meta.indexes` if not already present. The check is done with a single `first()` query — if None, skip.

The alert fires at most once per bot interaction per provider. Do NOT spam them on every message. Use the `last_interaction_at` timestamp as a proxy: only send the alert if the pending request is older than 1 hour (so it's not the same one they just responded to).

### 4. Customer GPS Collection

#### Flow

When a `TelegramUser` with `role=CUSTOMER` or `role=BOTH` (or unregistered customer default) tries to use a feature that requires proximity (swipe, grid, contact request creation), and `TelegramUser.has_customer_location` is False:

Intercept in the bot: before processing the action, send:
```
📍 To show you nearby service providers, we need your location.

Please tap the button below to share your GPS location:
```
With a ReplyKeyboard containing a location-share button (existing `request_location()` method in `TelegramBotService`).

Store the location in `TelegramUser.customer_latitude`, `TelegramUser.customer_longitude`, `TelegramUser.customer_location_updated_at`.

#### Ethiopia Bounding Box Validation

On receiving a GPS location from any user (provider or customer):

Ethiopia bounding box: latitude 3.4°N–15.0°N, longitude 33.0°E–48.0°E.

If the coordinates are outside this box:
- Send: "📍 Your location seems to be outside Ethiopia. Please re-share your GPS location from within Ethiopia."
- Include inline button: "🔄 Re-Share Location" → sends a new location-sharing keyboard.
- Do NOT save the out-of-range coordinates.

If the coordinates are inside Ethiopia but very far from Addis Ababa (Addis coords: lat=9.03, lon=38.74, distance > 600 km):
- Log a warning but ACCEPT the coordinates. Many Ethiopian cities are far from Addis.
- No user-facing message for this case.

#### Monthly Location Update Request

Once per month, for each provider with an approved service, if `ServiceProfile.location_update_requested_at` is null or older than 30 days:
- Send a Telegram message: "📍 Please confirm or update your service location. Tap below to share your current GPS location."
- With location-share keyboard.
- Set `ServiceProfile.location_update_requested_at = timezone.now()`.

This is triggered by cron endpoint `POST /api/admin/request-location-updates/` (admin auth required).
Returns `{"success": true, "sent_count": N}`.

### 7. Advertisement Broadcast

#### Admin endpoint `POST /api/admin/send-advertisement/`

- Auth: admin staff or admin TelegramUser.
- Algorithm:
  1. Fetch top 3 services by `likes_count DESC` where `approval_status=APPROVED` and `visibility_status=ON` and `admin_forced_hidden=False`.
  2. For each of these services, get their first photo (`order_index=1`).
  3. Also fetch the first service from each of the 3 most popular categories (by total approved services count).
  4. Deduplicate. Limit to 3 photos total.
  5. Send to all customers (role=CUSTOMER or BOTH) who have `policy_accepted_at IS NOT NULL`:
     - Send each photo using `TelegramBotService.send_photo(chat_id, photo_file_id, caption="...")`.
     - Caption: service title + category + "📍 " + city_text.
  6. Use a `ThreadPoolExecutor` (already exists in `approvals/contact_workflow.py`) for sending in parallel. Max 4 workers.
- Returns `{"success": true, "customers_targeted": N, "photos_sent": M}`.
- Log each send. Log failures.
- Rate limit: max 500 customers per advertisement call. Paginate if needed.

- Provider with exactly 10 requests, exactly 7 denials (70% ratio): no penalty.
- Provider with 10 requests, 8 denials (80% ratio): penalty applied.
- Customer with 3 saved services tries to save a 4th: returns 400.
- Survey callback for already-responded survey: idempotent update, no error.
